名前付きロックで実現する排他制御

Mysqlの名前付きロックについてとGoでの実装

toumakido

問題：SELECT FOR UPDATEの限界

在庫管理システムでの実装

トランザクション開始

商品の在庫をロックして取得 (SELECT FOR UPDATE)
if (在庫が存在する) {
    在庫を増やす (UPDATE)
} else {
    新しい在庫レコードを作成 (INSERT)
}

トランザクション終了

❌ レコードが存在しない場合、ロックできない

一般的な解決策とその課題

親テーブルのロック

課題：
  • 適切な親テーブルが存在しない場合がある
  • 関係のない処理まで待機させてしまう

初期値レコードの作成

課題：
  • ダミーレコードの管理が必要
  • 初期値を用意できない場合がある
→ レコードの存在に依存しない排他制御が必要！

名前付きロックによる解決

MySQLの名前付きロック（Named Lock）

  • 任意の文字列をキーとしてロックを取得・解放
  • データベースオブジェクトに依存しない

ロックの取得

SELECT GET_LOCK('product_ABC123', 10);

-- 戻り値:
-- 1=成功
-- 0=タイムアウト
-- NULL=エラー

ロックの解放

SELECT RELEASE_LOCK('product_ABC123');

-- 戻り値:
-- 1=成功
-- 0=ロック存在せず
-- NULL=エラー
✅ レコードの存在に関係なく確実にロックが取得できる！

Goで普通に実装すると...（ケース1）

// ロック取得
_, err := s.db.Exec("SELECT GET_LOCK(?, 10)", lockName)  // セッションA
if err != nil { return err }

// 実際の処理（SELECT, INSERT, UPDATE など）
// ...

// ロック解放
_, err = s.db.Exec("SELECT RELEASE_LOCK(?)", lockName)  // セッションB
return err

名前付きロックは、ロックを取得したセッションでしか解放できない

Goで普通に実装すると...（ケース2）

名前付きロックは、同一セッション内では同じロックを複数回取得できる（他セッションの場合に待機）

リクエスト1: GET_LOCK → プールからセッション100取得 → 成功
リクエスト2: GET_LOCK → プールからセッション100再利用
→ 同一セッションなので成功してしまう
→ 本来待機すべきリクエスト2が通る

リクエストごとに排他制御をする実装

conn, err := s.db.Conn(ctx) // 専用セッション(database/sql.Conn)を発行
if err != nil {
    return err
}

err = conn.GetLock(ctx, lockName) // 専用セッションでロック取得
if err != nil {
    return err
}

// トランザクション処理
tx, err := s.db.BeginTx(ctx)
... 実際の処理 ...

conn.ReleaseLock(ctx, lockName) // 専用セッションでロックリリース
conn.Close()

まとめ

✅ 名前付きロックの利点

  • レコードの存在に関係なく確実にロックが取得できる
  • 専用テーブルやダミーレコードが不要
  • ロック名を自由に設計できる

⚠️ Goでの実装の注意点

  • 専用コネクション（conn）の確保が必要
  • 同一セッション内でロック取得・解放を実行
